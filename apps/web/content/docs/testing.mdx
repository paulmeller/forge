## Test runner

Forge uses [Vitest](https://vitest.dev/) with explicit imports (no globals). Both `apps/web` and `apps/tick` have identical Vitest configs:

```ts
// vitest.config.ts
import { defineConfig } from 'vitest/config';

export default defineConfig({
  test: {
    globals: false,
    environment: 'node',
    setupFiles: ['./vitest.setup.ts'],
  },
});
```

`globals: false` means every test file must explicitly import from `vitest`:

```ts
import { describe, expect, it } from 'vitest';
```

## Running tests

```bash
# All packages
pnpm test

# Single package
pnpm --filter @forge/tick test
pnpm --filter @forge/web test

# Single file (from the package directory)
cd apps/tick && npx vitest run src/state.test.ts
```

Always run `pnpm -r test` before pushing.

## The "no mocking" philosophy

Forge tests avoid mocks, stubs, and spies. Instead, the codebase is structured so that core logic lives in **pure functions** that take plain data in and return plain data out. This makes tests:

- **Deterministic** -- same input always produces the same output.
- **Fast** -- no setup/teardown, no network, no database.
- **Resilient** -- refactoring internals doesn't break tests.

### Example: state machine

The `transition` function in `apps/tick/src/state.ts` is a pure function. Tests pass a status string and a synthetic event, then assert the returned state patch:

```ts
import { describe, expect, it } from 'vitest';
import { transition } from './state';

describe('transition', () => {
  it('dispatching тЖТ running on session.status_running', () => {
    const event = {
      id: 'sevt_test',
      type: 'session.status_running',
      processedAt: new Date(),
      raw: { type: 'session.status_running', id: 'sevt_fixture' },
    };
    const t = transition('dispatching', event);
    expect(t).toEqual({ status: 'running' });
  });
});
```

No HTTP client to mock, no database to seed, no adapter to stub.

### Example: budget computation

`computeBudgetPct` in `apps/tick/src/budgets.ts` takes a plain object and returns percentages:

```ts
it('computes token percentage correctly', () => {
  const result = computeBudgetPct({
    spentTokens: 800_000,
    budgetTokens: 1_000_000,
    budgetUsd: null,
  });
  expect(result.tokenPct).toBe(80);
  expect(result.maxPct).toBe(80);
});
```

### Example: adapter contracts

`apps/tick/src/adapters/adapter-contract.test.ts` verifies that both `ManagedAgentsAdapter` and `GatewayAdapter` implement the full `BackendAdapter` interface. It instantiates each adapter with dummy config (no real API calls) and asserts every required method exists:

```ts
function adapterSuite(name: string, create: () => BackendAdapter) {
  describe(`${name} contract`, () => {
    it('implements createSession', () => {
      expect(typeof create().createSession).toBe('function');
    });
    // ... sendTurn, listEvents, getSession, cancelSession, confirmToolUse
  });
}
```

## Writing new tests

### Where to put them

Test files live **next to the source file** they test, using the `.test.ts` suffix:

```
apps/tick/src/
тФЬтФАтФА state.ts
тФЬтФАтФА state.test.ts
тФЬтФАтФА budgets.ts
тФЬтФАтФА budgets.test.ts
тФЬтФАтФА adapters/
тФВ   тФЬтФАтФА types.ts
тФВ   тФЬтФАтФА managed-agents.ts
тФВ   тФЬтФАтФА gateway.ts
тФВ   тФФтФАтФА adapter-contract.test.ts
```

### How to structure them

1. **Extract a pure function** -- if the logic you want to test is tangled with I/O, pull it into a standalone function that takes data in and returns data out.
2. **Import from `vitest`** -- `import { describe, expect, it } from 'vitest';`
3. **Build plain fixtures inline** -- construct test data inside the test. No shared fixture files, no factory libraries.
4. **Assert return values** -- use `expect(...).toEqual(...)` or `expect(...).toBe(...)`. Avoid testing implementation details.

### Checklist for a new test

- [ ] File is named `<module>.test.ts` and sits next to `<module>.ts`
- [ ] Imports `describe`, `expect`, `it` from `vitest` (not globals)
- [ ] Tests pure functions with plain data -- no mocks, no stubs
- [ ] Passes when run in isolation: `npx vitest run src/<module>.test.ts`
- [ ] Passes as part of the full suite: `pnpm -r test`
