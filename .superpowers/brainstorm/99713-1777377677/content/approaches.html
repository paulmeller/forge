Three Landing Page Approaches

Same content, different structure and storytelling order

A

The Manifesto

Structure: Hero statement → Why open matters → 3 pillars (Ledger, Budgets, Adapters) → Use cases → GitHub CTA

Hero: "The open-source software factory." One sentence. Then a subline positioning against closed alternatives.

Feel: Like reading a declaration. Minimal sections, lots of whitespace, big typography. Each scroll reveals one idea. No feature grid — just three pillars explained in plain language.

Best for: The HN/Twitter visitor who decides in 5 seconds. Strong brand impression. Risks feeling light on substance for the evaluating eng manager.

B

The Architecture Story

Structure: Hero statement → "How it works" (Mission → Tasks → Gate → Merge) flow diagram → Three differentiators → Competitive comparison table → GitHub CTA

Hero: Same opening line, but followed immediately by a stylized architecture diagram showing the Mission lifecycle.

Feel: Like a well-designed technical README. Shows the system, not just the promise. Includes a "Forge vs." comparison table (open/closed, auditable/opaque, budgeted/unbounded).

Best for: The platform engineer evaluating tools. Gives them enough to decide without clicking into the repo. Risks feeling too "docs-y" for the casual visitor.

C

The Hybrid (Recommended)

Structure: Hero statement + single Console mock → 3 value props (short) → "How a Mission runs" (4-step visual) → "Why open source" section → GitHub CTA + quickstart

Hero: Bold statement left-aligned, hero screenshot/mock right-aligned. Clean split. GitHub stars badge visible.

Feel: Calm authority. Enough structure to be credible, enough restraint to feel premium. The console mock proves it's real. The value props are one-liners, not essays. The mission flow is visual, not textual.

Best for: All three audiences. The casual visitor gets the headline. The engineer gets the flow. The leader gets the "why open" argument. Each section works independently.
